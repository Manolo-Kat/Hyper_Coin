main.py — 6 fixes

utils/db.py — 7 fixes

8. The database schema has no indexes. Add indexes on: the guild_id column of the users table, the guild_id + coins columns together on the users table for leaderboard queries, the guild_id + user_id columns on daily_earnings, the guild_id + user_id columns on weekly_spending, and the guild_id + user_id columns on daily_claims. These indexes must be created with IF NOT EXISTS so they are safe to add to an already running database.

9. The adjust_coins() function currently reads the balance first and then writes back the new value as two separate operations. This is a race condition — if two coin changes happen at the same time for the same user, one of them will be silently lost. Replace the entire read-then-write with a single atomic SQL statement that does the addition and the MAX(0, ...) floor in one operation. Remove the separate get_balance() call from this function.

10. get_guild_config() hits the database on every single command and on every single message the bot receives. It has no caching at all. Add a simple in-memory cache at module level with a 60-second TTL. When set_guild_config_field() is called, invalidate the cache for that guild so the next read gets fresh data.

11. The make_week_key() function embeds the user_id inside the week key string even though user_id is already stored as a separate column in the weekly_spending table. This is redundant and wastes space. Change the function to return only the year and week number. The table's primary key already has user_id as its own column so this change is safe. Update every call site of this function accordingly.

12. The add_earned_coins() function calls db.commit() twice inside the same function — once after each INSERT. Consolidate to a single commit after both INSERTs.

13. The get_leaderboard() query fetches every single user row in the guild with no limit. Add a LIMIT of 100 rows since the leaderboard only ever displays 10 at a time.

14. Add a background task (start it in on_start in main.py) that runs every hour and executes PRAGMA wal_checkpoint(PASSIVE) and PRAGMA optimize on the database connection. This prevents the WAL file from growing unbounded on disk.

utils/helpers.py — 4 fixes

15. The get_exchange_rate() function has a race condition. It acquires the rate lock, checks the cache, releases the lock, and then makes the HTTP request. Two coroutines can both see an empty cache simultaneously and both fire HTTP requests at the same time. Restructure the function so the HTTP fetch also happens while the lock is held, preventing duplicate in-flight requests.

16. The SpamTracker._penalty dictionary is checked for expired entries on access but the expired entries are never deleted. Verify that when an expired penalty is found, the entry is deleted from the dictionary so it does not accumulate forever.

17. The SpamTracker similarity check runs difflib.SequenceMatcher on every message. This is computationally expensive for long messages. Skip the similarity check entirely for any message longer than 150 characters since long messages are almost never spam repetitions.

18. Both admin.py and shop.py have their own private copy of the same log-sending helper function (_log and _log_embed respectively). They are identical except for the name. Move this into utils/helpers.py as a single shared function called send_log_embed. Update both admin.py and shop.py to import and use it. The current versions silently swallow all exceptions — the shared version must log a warning when sending fails.

extensions/economy.py — 8 fixes

21. In daily_cmd, the random coin amount is being multiplied by 2 for boosters before being passed to add_earned_coins(). But add_earned_coins() already applies the booster 2x multiplier internally. This means boosters are getting 4x coins from daily instead of 2x. Remove the pre-multiplication in daily_cmd and let add_earned_coins() handle the booster multiplier on its own.

22. In daily_cmd (and anywhere else this pattern appears in economy.py), the streak is read with a check that treats a streak of 0 as falsy and falls back to 0. This is coincidentally correct but the pattern is fragile. Change all such checks to explicitly test against None instead of relying on truthiness.

23. The daily_cmd command checks if the user is banned but does not check the allowed_roles list. A user who is blocked from earning coins via chat can still claim the daily reward. Add the same allowed_roles check that the on_message listener already performs.

24. The balance_cmd command always calls bot.rest.fetch_member() to get the target member — even when the target is the command invoker themselves, who is already available from ctx.member at no cost. If the invoker is checking their own balance, use ctx.member directly. If checking another user and fetch_member raises NotFoundError, respond with a clear message instead of silently defaulting to non-booster status.

25. When a user types 'coins' in the currency command to reset their display currency, the code stores the string 'USD' in the database. But balance_cmd checks whether the stored value is 'COINS' to decide whether to show coins instead of a converted value. Since 'COINS' is never actually stored, this check never triggers and the feature is broken. Fix by storing the literal string 'COINS' in the database on reset instead of 'USD'.

26. The LeaderboardView class has no on_timeout method. When the 60-second timeout fires, miru calls the default which does nothing and the view stays referenced in memory. Add an on_timeout method that calls self.stop().

27. Any user can click the pagination buttons on any leaderboard that someone else summoned. Store the author's user ID when the view is created and in the button handler check that the person clicking is the same person who ran the command. 28. In the on_message listener, get_guild_config() is called on every message before the cooldown and spam checks run. This means even messages that will be immediately rejected (because of cooldown) still cost a database query. Move the spam detector check and the cooldown check to the very top of the function, before any database calls.
