TASK 1 — Stop saving the whole file on every single message
Difficulty : EASY
Tags       : perf
Agreed by  : All 4 sources (You, Gemini, DeepSeek, ChatGPT)

Right now save_data() gets called every time anyone sends a message
because add_coins() calls it. On a busy server that's hundreds of
full file rewrites per minute. It's the #1 performance problem.

The fix: use asyncio.Event() as a dirty flag. When data changes,
call event.set(). A background loop waits for that signal, saves
the file, then clears the flag. This reacts instantly to changes
but still batches rapid writes — better than a plain sleep(30) loop.

    save_event = asyncio.Event()

    def mark_dirty():
        save_event.set()

    async def autosave_loop():
        while True:
            await save_event.wait()   # waits until mark_dirty() is called
            save_event.clear()
            await asyncio.sleep(5)    # small delay to batch rapid changes
            await save_data()

    # Start this in on_start():
    asyncio.create_task(autosave_loop())

Now replace almost every "await save_data()" with "mark_dirty()".
Only keep immediate saves for purchases and admin coin edits
because those are high-stakes and need to persist right away.

TASK 2 — Stop creating a new internet connection on every command
Difficulty : EASY
Tags       : perf
Agreed by  : You, ChatGPT

Every time someone runs /balance or /buy, the bot opens a brand new
aiohttp session, uses it once, and throws it away. That's wasteful.

The fix: make one session when the bot starts and reuse it everywhere.

    http_session = None

    @bot.listen(hikari.StartedEvent)
    async def on_start(_):
        global http_session
        http_session = aiohttp.ClientSession()
        load_data()
        asyncio.create_task(autosave_loop())

    @bot.listen(hikari.StoppingEvent)
    async def on_stop(_):
        await http_session.close()

Then replace every "async with aiohttp.ClientSession() as s:" with
just "async with http_session.get(...) as r:".

TASK 3 — Cache currency exchange rates instead of fetching live every time
Difficulty : EASY
Tags       : perf
Agreed by  : All 4 sources

Right now every /balance call hits the currency API. If 10 people
check their balance at the same time that's 10 separate web requests.
Exchange rates don't change every second so there's no reason for this.

The fix: store the rate in memory with a timestamp. If the stored rate
is less than 60 minutes old, use it. Only fetch from the API when it's
actually stale. Use asyncio.Lock so two commands don't both fetch at
the same time when the cache is empty.

    rate_cache = {}
    rate_lock = asyncio.Lock()

    async def get_exchange_rate(currency):
        async with rate_lock:
            if currency in rate_cache:
                rate, fetched_at = rate_cache[currency]
                age = (datetime.now(timezone.utc) - fetched_at).seconds
                if age < 3600:
                    return rate
        async with http_session.get(
            f"https://api.frankfurter.dev/v1/latest?base=USD&symbols={currency}"
        ) as r:
            data = await r.json()
            rate = data['rates'].get(currency)
            rate_cache[currency] = (rate, datetime.now(timezone.utc))
            return rate

Replace all the inline currency fetching in /balance and /buy
with a call to this function.

TASK 4 — Add a cooldown so messages don't earn coins every single time
Difficulty : EASY
Tags       : security, perf
Agreed by  : You, ChatGPT

Right now someone could spam short messages and earn coins extremely
fast. The bot also calls add_coins() for literally every message sent
in the server which hammers it unnecessarily.

The fix: track when each user last earned coins from a message.
Only award coins if at least 60 seconds have passed. Also optionally
skip messages shorter than 5 characters.

    message_cooldowns = {}

    @bot.listen(hikari.GuildMessageCreateEvent)
    async def on_message(event):
        if event.is_bot or not event.guild_id:
            return
        if len(event.content or "") < 5:
            return
        now = datetime.now(timezone.utc)
        last = message_cooldowns.get(event.author_id)
        if last and (now - last).total_seconds() < 60:
            return
        message_cooldowns[event.author_id] = now
        gdata = get_guild_data(event.guild_id)
        if event.channel_id in gdata['uncounted']:
            return
        if is_banned(event.guild_id, event.member):
            return
        await add_coins(event.guild_id, event.author_id, 1, event.member)

TASK 18 — Add a $20 per week purchase limit [NEW]
Difficulty : MEDIUM
Tags       : feature, security

Users should have a maximum of $20 worth of purchases per week.
When they hit the limit they can't buy again until Monday resets it.

Important: always store the week key as "userid_year_weeknumber"
not just "userid_weeknumber". Without the year, week 1 of 2025 and
week 1 of 2026 look identical in the dictionary and clash.

    from datetime import timedelta

    now = datetime.now(timezone.utc)
    iso = now.isocalendar()
    week_key = f"{ctx.user.id}_{iso.year}_{iso.week}"

    weekly_spent = gdata.setdefault('weekly_spent', {})
    spent_this_week = weekly_spent.get(week_key, 0)

    if spent_this_week + ctx.options.amount > 20:
        days_until_monday = (7 - now.weekday()) % 7 or 7
        reset_time = now + timedelta(days=days_until_monday)
        reset_time = reset_time.replace(hour=0, minute=0, second=0, microsecond=0)
        reset_ts = int(reset_time.timestamp())
        await ctx.respond(
            f"You've hit the $20/week limit (${spent_this_week} spent).\n"
            f"Resets <t:{reset_ts}:R>.",
            flags=hikari.MessageFlag.EPHEMERAL
        )
        return

    # After purchase goes through:
    weekly_spent[week_key] = spent_this_week + ctx.options.amount
    mark_dirty()

TASK 20 — Approval buttons stop working after the bot restarts [NEW]
Difficulty : HARD
Tags       : bugfix, feature

This is a big one. When the bot restarts, all the miru Views that
were attached to pending purchase messages are gone from memory.
The buttons are still visible in Discord but clicking them does nothing.

The fix is to store every pending purchase in data.json when it's
created, then re-attach a new ShopApprovalView to each one on startup.

Step 1 — when a purchase request is posted, save it:
    gdata.setdefault('pending_purchases', {})[str(message_id)] = {
        'user_id': ctx.user.id,
        'type': ctx.options.type,
        'amount': ctx.options.amount,
        'coins': req,
        'channel_id': approval_channel_id,
        'message_id': message_id
    }
    mark_dirty()

Step 2 — in on_start(), after load_data(), loop through all
pending purchases and re-attach views to the existing messages:
    for gid_str, gdata in guild_data.items():
        gid = int(gid_str)
        for msg_id_str, p in gdata.get('pending_purchases', {}).items():
            view = ShopApprovalView(
                gid,
                p['user_id'],
                p['type'],
                p['amount'],
                p['coins']
            )
            miru_client.start_view(view, bind_to=int(msg_id_str))

Step 3 — in the approve and reject callbacks, remove the entry:
    gdata.get('pending_purchases', {}).pop(str(self.message_id), None)
    mark_dirty()

This works because miru can bind a new View to an existing Discord
message by its message ID — the buttons are already there, you're
just re-attaching the logic to them.

TASK 22 — Switch from data.json to a real database (SQLite)
Difficulty : HARD
Tags       : scale, perf
Agreed by  : All 4 sources

The JSON approach works fine for now but will become a problem as
the server grows. SQLite is a lightweight database that's built into
Python — no server needed, no extra setup beyond installing aiosqlite.

Why it's better than JSON:
  - Only updates the one row that changed (not the whole file)
  - Handles concurrent reads/writes safely (eliminates the need for asyncio.Lock in Task 9)
  - Leaderboard becomes one SQL query instead of sorting everything in Python
  - Weekly limits (Task 18), daily earnings (Task 6), and pending purchases (Task 20)
    all become simple database queries instead of nested dicts

This also makes Tasks 6, 7, 18, and 20 much simpler to implement.
Do this after everything else is working.

Install: pip install aiosqlite
